# Архітектура проєкту

## Принцип на старті

MVP будується як модульний моноліт: окремий frontend і один backend API з
чіткими предметними модулями. Це дешевше й швидше за мікросервіси, але не
блокує подальше масштабування.

```text
Telegram / Browser
       │
       ▼
Frontend (Next.js / React / TypeScript)
       │ REST / JSON
       ▼
Backend (NestJS / Node.js)
       ├── auth + Telegram initData
       ├── context + QR
       ├── places + search
       ├── booking + resources
       ├── transactions + bonuses
       ├── ledger + settlement
       ├── reviews + moderation
       ├── ambassadors
       └── analytics + audit
       │
       ├── PostgreSQL + PostGIS
       ├── Redis / background jobs
       └── S3-compatible storage
```

На етапі 1 запущені frontend, healthcheck і read-only project endpoints. База
даних, Redis і сховище навмисно не підключені до появи реальної бізнес-логіки.
Для Railway frontend збирається стандартними `next build` / `next start`;
Vinext використовується лише як адаптер середовища інтерактивного preview.

## Рівні доступу

Доступ перевіряється на backend, а не тільки приховується в інтерфейсі:

1. платформа;
2. регіон / tenant;
3. організація партнера;
4. конкретний заклад;
5. конкретний ресурс або зміна.

Регіональний адміністратор не повинен отримувати дані іншого tenant навіть при
ручній підміні URL або параметра API.

## QR

- статичний QR точки входу визначає контекст, але не фізичну присутність;
- динамічний QR клієнта містить signed token, nonce та expiry 30–60 секунд;
- токен одноразовий;
- персональні дані у відкритому QR відсутні;
- Telegram initData перевіряється сервером;
- операції використовують idempotency key.

## Бронювання кількох ресурсів

`Place` має багато `Resource`. Кожен ресурс має свій графік, тривалість слота,
buffer, місткість, ціну і правила доступності. Ручний запис партнера блокує слот
так само, як клієнтське бронювання. Подвійне бронювання блокується транзакційно
на рівні бази даних.

## Бонуси й фінанси

Баланс не є полем, яке можна змінити вручну. Він обчислюється з незмінних
`LedgerEntry`. Refund, cancellation або виправлення створюють окрему reversal
операцію з посиланням на початкову. Ставки й правила зберігаються snapshot на
момент транзакції.

## Середовища

- local — розробка;
- staging — тестовий сервер і приймання етапів;
- production — бойове середовище після етапу 7.

Секрети не зберігаються у Git. Приклади змінних наведено у `.env.example` та
`backend/.env.example`.
